# H2A MCP — Specification 1.0

Human Interface to Agentic Interface. A specification for re-describing products designed for human operation — websites, desktop and mobile applications, REST surfaces, and other interactive media — as structured tool calls consumable by agents via the Model Context Protocol (MCP).

This document is the normative definition of the four artifacts and the rules that bind them. The accompanying JSON Schemas under `schema/1.0/` are the machine-readable form of these rules; where prose and schema disagree, the schema governs for structural validation and this document governs for intent. See `README.md` for the conceptual overview and motivation.

## 1. Scope and version

H2A maps a human interface onto an agentic interface: it wraps an interface originally built for human use so an agent can operate it programmatically. The central property: human-facing interfaces change continually, and H2A confines that volatility to the binding layer, beneath a stable, versioned, agent-facing tool surface.

This is spec version **1.0**. The version line is shared across all four schemas in `schema/1.0/` and this document (`spec/1.0.md`). A future revision is a new `schema/1.1/` set plus `spec/1.1.md`; existing instances keep pointing at the `1.0` schemas they were authored against.

## 2. Artifacts

H2A defines four artifacts. Three are instance artifacts that describe real systems; the fourth (`surface`) is a shared controlled vocabulary.

| Artifact     | Scope                          | Defines                                                          | Schema                      | Consumed by        |
| ------------ | ------------------------------ | --------------------------------------------------------------- | --------------------------- | ------------------ |
| **Contract** | Shared, per category           | The abstract tool interface: methods + param/return/error schemas | `schema/1.0/contract.json` | Agent + Binding    |
| **Manifest** | Per (org × surface)            | Which contracts, which methods, and which binding version are live | `schema/1.0/manifest.json` | Agent              |
| **Binding**  | Per (org × surface × contract) | The recording that implements each method                       | `schema/1.0/binding.json`  | Orchestrator only  |
| **Surface**  | Global vocabulary              | The closed set of interaction surfaces                          | `schema/1.0/surface.json`  | Manifest + Binding |

The defining property: **contracts are abstract and shared; manifests and bindings are concrete and partitioned by org and surface.** A site redesign is a binding-only republish — no contract change, nothing agent-visible changes.

## 3. Repository layout

```
spec/1.0.md
schema/
  1.0/
    contract.json        # meta-schema: contract artifacts
    manifest.json        # meta-schema: manifest artifacts
    binding.json         # meta-schema: binding (recording) artifacts
    surface.json         # the surface vocabulary [www, app, rest]
contract/
  banking@1.2.json       # shared, abstract; versioned filename so versions coexist
org/
  <org>/
    <surface>/
      manifest.json                 # the (org × surface) manifest
      <contract>@<bindingVersion>.json   # the binding/recording
```

Worked example in this repo:

```
contract/banking@1.2.json
org/example/www/manifest.json
org/example/www/banking@1.5.json
```

Naming rules:

- **Contract:** `contract/<id>@<version>.json`. The filename stem before `@` equals `id`; the `@<version>` equals `version`. Versioned filenames let multiple contract versions sit on disk at once — required because manifests pin exact versions and migration is deliberate (§7).
- **Manifest:** `org/<org>/<surface>/manifest.json`. Org is the parent dimension; surface is a folder beneath it. One manifest per (org × surface).
- **Binding:** `org/<org>/<surface>/<contract>@<bindingVersion>.json`. The `@<version>` is the *binding* version, not the contract version; the contract version it targets is recorded inside as `satisfiesContract`.

## 4. Contract — the shared interface

A contract is the abstract vocabulary for a category (e.g. `banking`). It defines the full menu of methods; each method declares `params`, `returns`, and a closed set of `errors`. It is authored once and reused across every org and surface that implements the category. Param and return schemas use a constrained JSON-Schema subset (a `type`, plus `properties`/`required` for objects).

Two derived guarantees the rest of the spec leans on:

- The **param** property names are the only valid `{{tokens}}` a binding may inject.
- The **return** property names are the only valid output fields a binding may extract, and the **error** codes are the only codes a binding may raise.

A contract may declare an optional `required` array — a conformance core of methods every implementing manifest must include (§6).

## 5. Surface — the interaction vocabulary

A surface names the medium an agent operates against. Spec 1.0 defines a closed set: `www` (browser DOM), `app` (native application), `rest` (HTTP API). The surface is the second partition axis after org: a single org may publish independent manifests and bindings per surface, because the same contract method is reached differently on each. New surfaces are added by extending `schema/1.0/surface.json`.

## 6. Manifest — the per-(org × surface) declaration

A manifest names an org and a surface, and lists the contracts live on that surface. For each contract it pins:

- `version` — the exact contract version (e.g. `1.2`).
- `binding` — the exact binding version to load (e.g. `1.5`).
- `methods` — the subset of the contract's methods that are actually live.

The manifest's declared `methods` are the **single source of truth for what is callable**. A method the manifest omits never appears to the agent, even if the contract defines it and a binding implements it.

```jsonc
// org/example/www/manifest.json
{
  "kind": "manifest",
  "org": "example",
  "surface": "www",
  "version": "1.0",
  "entryPoint": "example.com",
  "contracts": [
    { "id": "banking", "version": "1.2", "binding": "1.5",
      "methods": ["checkBalance", "transfer", "payBill"] } // no wireTransfer / orderChecks
  ]
}
```

## 7. Binding — the recording

A binding implements a contract's methods for one (org × surface). It is **one-and-done**: authored by recording the in-page input, output, and actions for each method. It is private to the orchestrator and never seen by the agent.

Each method is a `steps` array plus an optional `errors` map. The step vocabulary is semantic, not raw DOM events: `navigate`, `click`, `input`, `select`, `submit`, `waitFor`, `assert`, `read`.

**Locators are multi-strategy and ordered.** Every targeted step carries a `locators` array tried in priority order until one resolves uniquely — from most resilient (`role`, `testid`, `label`) to most brittle (`css`, `xpath`). This ordered redundancy is what lets a binding survive small redesigns and, when it can't, what a re-recording replaces.

**The bind map is explicit and two-sided.**

- *Inputs:* `input`/`select` steps inject contract params via `{{paramName}}` tokens in `value`. Only names from the method's contract `params` are valid.
- *Outputs:* `read` steps carry an `extract` that maps a captured value to a named `field` of the method's contract `returns`, with optional `as` coercion.

**Synchronization is explicit.** A `waitFor` step (or a `wait` pre-condition on any step) holds on a `selector`, `selectorGone`, `url`, `networkIdle`, or `timeout` condition before proceeding. A condition that exceeds its `timeoutMs` is a failure.

**Errors map to the contract.** The `errors` array maps a recognized failure condition (`selectorVisible`, `textContains`, `stepTimeout`, `urlMatches`) to a contract error `code`. A binding may only raise codes the contract declares for that method.

```jsonc
// org/example/www/banking@1.5.json (excerpt)
{
  "kind": "binding", "org": "example", "surface": "www",
  "contract": "banking", "satisfiesContract": "1.2", "version": "1.5",
  "methods": {
    "transfer": {
      "steps": [
        { "action": "navigate", "url": "https://www.example.com/transfer",
          "wait": { "type": "networkIdle", "timeoutMs": 15000 } },
        { "action": "input", "value": "{{amount}}",
          "target": { "locators": [
            { "strategy": "label", "value": "Amount" },
            { "strategy": "testid", "value": "transfer-amount" } ] } },
        { "action": "read", "extract": { "field": "confirmationId", "from": "text", "as": "string" },
          "target": { "locators": [ { "strategy": "testid", "value": "confirmation-number" } ] } }
      ],
      "errors": [
        { "when": { "selectorVisible": { "locators": [ { "strategy": "testid", "value": "error-insufficient-funds" } ] } },
          "raise": "INSUFFICIENT_FUNDS" }
      ]
    }
  }
}
```

## 8. Agent / Orchestrator split

Two planes, with the contract as the boundary.

```
Agent-facing plane (discovery + invocation)        Orchestration plane (execution)
-------------------------------------------        -------------------------------
needs: Manifest + Contracts                        needs: Manifest + Bindings
sees:  which tools exist + their schemas           holds: the recordings (private)
emits: example.www.banking.transfer(x,y,z)  →   resolves binding, runs steps
```

- The **agent** needs the **manifest + contracts**. It resolves the referenced contracts, intersects them with the manifest's declared methods, and exposes the live tools — the contract is what defines each tool's name and its param/return/error schemas. The agent validates call arguments against the contract `params` before emitting. It never touches bindings — recording internals are irrelevant to it and a leak risk.
- The **orchestrator** needs the **manifest + bindings**. It receives the resolved, fully-qualified call, uses the manifest to look up the binding keyed by `(org, surface, contract, binding@version, method)`, binds the supplied arguments into the recording's `{{tokens}}`, runs the recorded steps, and returns the extracted `returns` (or a mapped error). It does not need the contract at runtime: the binding is self-contained, and argument validation has already happened agent-side.

The contract is needed by the **agent** at runtime and by the **validator** at publish time (§12); it is not a runtime dependency of the orchestrator.

Because bindings are an implementation detail, they can be private, swapped freely, or offered in multiple flavors — all invisible to the agent.

## 9. Fully-qualified tool name

The agent emits calls under the keypath:

```
<org>.<surface>.<contract>.<method>      e.g.  example.www.banking.transfer
```

The surface is part of the name because the same contract method on a different surface is a different live tool with a different binding.

## 10. Resolution flow

```
Manifest            →   Contracts           →   Binding
(which org+surface)     (which tools)            (how to execute)
```

1. Agent asks about `example` on `www` → fetch `org/example/www/manifest.json` → it declares `[banking@1.2 via binding 1.5]`.
2. Agent resolves contract `banking@1.2` → intersect with the manifest's declared methods → tool list is `checkBalance`, `transfer`, `payBill`.
3. Agent calls `transfer` → emits `example.www.banking.transfer(from, to, amount)`.
4. Orchestrator loads binding `org/example/www/banking@1.5.json`, confirms `satisfiesContract == 1.2`, validates args, runs the `transfer` recording, extracts `confirmationId` + `status` (or raises a mapped error).

Two axes: *resolution order* runs manifest → contract → binding; *abstraction* puts the contract above both as the shared interface the manifest declares and the binding implements.

## 11. Versioning

All artifacts version independently because each drifts for a different reason. Every artifact uses a single `version` field; the binding additionally records `satisfiesContract`.

| Artifact     | Version line                 | Bumps when                                                       |
| ------------ | ---------------------------- | ---------------------------------------------------------------- |
| **Contract** | `banking@1.2`                | A method or param/return/error schema is added, changed, renamed |
| **Manifest** | `example/www@1.0`         | Declared support changes, or a contract/binding pin moves        |
| **Binding**  | `example/www/banking@1.5` | The recording changes (redesign, relocator, flow fix)            |

Rules:

- **The contract version is the contract between manifest and binding.** A manifest pins an exact contract version; a binding records the version it satisfies. The two must match.
- **The binding version is free to move under a fixed contract version.** A relocator ships binding `1.5 → 1.6` while still satisfying `banking@1.2`. This is the whole payoff: a redesign is a binding-only republish.
- **Pins are exact, not ranges.** The orchestrator must have an exact matching binding, so exact pins keep resolution predictable.
- **Nothing auto-upgrades.** When a contract bumps `1.2 → 1.3`, a manifest stays on `1.2` until a `1.3` binding is authored, verified, and the manifest is explicitly re-pinned. Migration is deliberate and atomic per (org × surface). Versioned contract filenames let both versions coexist during the migration window.

## 12. Validation invariants

Checked at publish time, so the artifacts stay in sync by validation rather than convention. For each manifest contract entry `m` referencing contract `c@m.version` and binding `b`:

1. **Method subset:** `m.methods ⊆ c.methods` — a manifest can't claim methods the contract doesn't define.
2. **Conformance floor (if present):** `c.required ⊆ m.methods` — a manifest can't claim a category while omitting its required core.
3. **Binding coverage:** `keys(b.methods) == m.methods` — every declared method has a recording; no orphan recordings.
4. **Contract pin match:** `b.satisfiesContract == m.version` — catches "contract moved, binding didn't."
5. **Resolvable contract:** `c@m.version` resolves to a real published contract file.
6. **Bind-map integrity:** every `{{token}}` in `b` names a property of the method's contract `params`; every `extract.field` names a property of the method's contract `returns`; every `errors[].raise` is one of the method's contract error `codes`.
7. **Path/identity agreement:** an artifact's `org`/`surface`/`id`/`version` fields agree with its location and filename.

## 13. Authoring summary

Per (org × surface), the minimum authoring set is **two documents**:

- **1 manifest** — declares contracts + method subsets + version pins.
- **N bindings** — one recording per supported contract.

Contracts and surfaces are shared: authored once and reused across all orgs and surfaces. At runtime the agent needs *manifest + contracts* (to define the tool API) and the orchestrator needs *manifest + bindings* (to execute) — both pivot on the manifest, neither needs the other's private half. The contract is the published interface; the binding is a private recording; the manifest is where per-(org × surface) partiality lives. (Validation at publish time needs all three together — §12.)
